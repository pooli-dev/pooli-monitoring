  실행 계획

  보내주신 방향은 대체로 맞지만, 이미 있는 것을 다시 만들 필요는 없습니다. 실제로 해야 할 작업만 정리합니다.

  Phase 1: 로그 노이즈/보안 정리 (즉시)

  작업 1-1: 필터 통합 — LoggingFilter + RequestLoggingFilter를 하나로 합치기

  현재 문제:
  - 두 필터가 각각 traceId를 생성해서 UUID가 겹침
  - LoggingFilter가 raw body를 찍고, RequestLoggingFilter는 마스킹된 body를 캐시만 함
  - 역할이 분리되어 있지만 실제로는 하나의 흐름

  해야 할 것:
  - RequestLoggingFilter를 삭제하고 LoggingFilter 하나로 통합
  - traceId 생성은 한 번만
  - body 마스킹 로직을 LoggingFilter에 통합
  - 로그 출력 시 마스킹된 body 사용

  작업 1-2: /api/auth/ 요청은 body 로깅 금지**

  - 로그인, 회원가입 등 인증 관련 API는 body에 password, token이 들어있음
  - URI가 /api/auth/로 시작하면 body를 [REDACTED]로 대체

  작업 1-3: 민감 필드 마스킹 확대

  현재 password만 마스킹 중. 추가 필요:
  - token, accessToken, refreshToken
  - authorization
  - cookie
  - secret, credential

  작업 1-4: LoggingFilter에서 body 기본 제거, 에러 시에만 기록

  - 정상 요청(2xx, 3xx): body 로그 안 남김
  - 에러 요청(4xx, 5xx): 마스킹된 body만 남김
  - 이유: 정상 요청의 body는 RCA에 필요 없고, 로그량만 늘림

  작업 1-5: logback-spring.xml 운영 프로필 appender 추가

  현재 !local 프로필에 appender가 없음. Docker의 stdout으로 JSON 로그가 나가야 Promtail이 수집 가능.

  수정:
  <springProfile name="!local">
      <root level="INFO">
          <appender-ref ref="CONSOLE"/>
      </root>
  </springProfile>

  Phase 2: 요청 로그 표준화 (Phase 1 완료 후)

  작업 2-1: LoggingFilter 로그 포맷 정리

  현재:
  apiUri=/api/... method=POST status=200 latency=123ms body={...}

  변경 후:
  apiUri=/api/... method=POST status=200 latency=123ms
  (body는 에러 시에만, 별도 필드로)

  이미 JSON 형식(LogstashEncoder)이므로 MDC에 아래 필드를 추가하면 구조화된 검색이 가능:
  - traceId (이미 있음)
  - apiUri
  - method
  - status
  - latency

  작업 2-2: GlobalExceptionHandler 로그에서 body 제거

  handleApplicationException에서 body를 log.error의 마지막 인자로 넘기고 있음 (line 49). 포맷 문자열에 {}가 4개인데 인자가 5개라 body가 Throwable로 인식될 수 있음. body를 제거하고 traceId로 연결하는 방식으로 변경.

  Phase 3: DB/Redis 구간 메트릭 (Phase 2 완료 후)

  이게 RCA의 핵심입니다.

  Spring Boot의 http.server.requests로 이미 endpoint별 latency/error/rate를 Prometheus PromQL만으로 구할 수 있습니다. 별도 커스텀 메트릭은 불필요합니다.

  하지만 "그 latency 중에서 DB가 얼마, Redis가 얼마"를 알려면 추가 코드가 필요합니다.

  작업 3-1: MyBatis 쿼리 실행 시간 측정 (Interceptor)

  - MyBatis Interceptor를 만들어서 각 쿼리의 실행 시간을 Micrometer Timer로 기록
  - 태그: mapper (예: UserMapper), operation (예: selectById)
  - 메트릭명: mybatis_query_duration_seconds
  - 절대로 raw SQL을 태그로 넣으면 안 됨 (카디널리티 폭발)

  작업 3-2: Redis 호출 시간 측정

  - Spring Boot의 Lettuce 자동 설정에 Micrometer 연동 가능
  - spring.data.redis.lettuce.metrics.enabled=true 또는 커스텀 RedisTemplate wrapper
  - 메트릭명: redis_command_duration_seconds
  - 태그: command (GET, SET, HGET 등)

  작업 3-3: metrics_contract.yaml 업데이트 (pooli-monitoring 쪽)

  pooli-monitoring의 metrics_contract.yaml에 현재 traffic 메트릭만 있음. 새로 추가되는 DB/Redis 메트릭을 계약에 추가해야 anomaly detection 대상에 포함됨.

  Phase 4: pooli-monitoring 연동 (Phase 3 완료 후)

  작업 4-1: Grafana 대시보드에 API 패널 추가

  http.server.requests 기반 PromQL:
  - rate(http_server_requests_seconds_count{...}[5m]) → endpoint별 RPS
  - histogram_quantile(0.95, rate(http_server_requests_seconds_bucket{...}[5m])) → P95 latency
  - rate(http_server_requests_seconds_count{status=~"5.."}[5m]) → error rate

  작업 4-2: DB/Redis 메트릭 대시보드 추가

  - mybatis_query_duration_seconds P95 by mapper
  - redis_command_duration_seconds P95 by command

  작업 4-3: Loki 로그 기반 알림 업데이트

  현재 ERROR/FATAL/WARN 로그 수만 카운트. error_type별 분류 알림 추가 가능.

  한 줄 결론

  보내주신 분석의 방향은 맞습니다. 다만 endpoint 메트릭은 Spring Boot가 이미 제공하므로 직접 구현할 필요 없고, 실제로 코드 작업이 필요한 건 필터 정리(보안+노이즈), DB/Redis 구간 메트릭 추가, logback 운영 설정 수정 이 세 가지가  
  핵심입니다.
